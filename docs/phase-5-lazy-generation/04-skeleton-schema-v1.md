# Phase 5.4.1 — Skeleton Schema (Locked)

**Status:** Locked design. Implementation begins in Phase 5.4.2.
**Schema version:** 5.4.1

## Purpose

Defines the JSON structure that Sonnet (orchestrator) produces and Haiku (worker) consumes. This is the contract between the two models. Once this schema is locked, the Sonnet skeleton prompt, the Haiku fan-out dispatcher, the slot resolver, and the player rendering layer all build against it.

## Design principles

**Slot-shaped, not text-shaped.** Sonnet emits IDs, labels, flags, and slot references. Haiku fills in the prose. This separation is what enables parallel fan-out — Haiku can generate slot 17 without knowing anything about slot 18.

**Phases are an array, not fixed.** The schema supports n phases. v1 ships with 2 (assess + intervene), but the structure accommodates future multi-round and curveball without refactoring.

**Stage type drives behavior.** Each phase has a stageType field that the player and the dispatcher both branch on. Adding a new stage type later (e.g., triage, consult) is a one-line addition to the type union.

**Wave-based fan-out is implicit in phase order.** Phases generate in priority order (Wave N corresponds to phase N's content). No explicit priority field needed — array order is the priority.

**Slot refs use a stable address scheme.** Every fillable slot has a _slotRef that uniquely identifies it. This is what fetchExplanations writes back into.

## Top-level structure

The scenario object has three groups of fields:

**Static metadata** — schemaVersion, id, source, createdAt. Not generated by Sonnet beyond id (which Sonnet picks); the rest are added by the client.

**Sonnet-authored intro content** — title, subtitle, patientCard, presentation. This is the EMS-report screen. Generated fully by Sonnet in the same call as the skeleton because the user reads it first while Wave 1 fires in the background. This block precedes Phase 1 conceptually and contains the framing the user needs before assessment begins.

**Skeleton arrays** — phases, debrief. These contain the slot-shaped structure that drives Haiku fan-out.

Example top-level shape:
{
"schemaVersion": "5.4.1",
"id": "septic-shock-6yo",
"source": "ai",
"createdAt": "2026-05-07T15:30:00Z",
"title": "...",
"subtitle": "...",
"patientCard": { ... },
"presentation": { ... },
"phases": [ ... ],
"debrief": { ... },
"_meta": { ... }
}

## Phase structure

Each phase in the phases array has the shape:
{
"phaseIndex": 0,
"stageType": "assess",
"title": "Initial Assessment",
"narrative": "...",
"vitals": [ ... ],
"signs": [ ... ],
"labs": [ ... ],
"actions": {
"tools": { ... },
"meds": { ... }
}
}

**stageType values:** "assess", "intervene", "reassess", "curveball", "debrief". v1 uses "assess" and "intervene" only.

**narrative:** Sonnet writes this directly in the skeleton call. It is the framing prose the user reads when entering the phase ("EMS arrives," "After IV access and fluids, you reassess..."). Not a slot — Sonnet authors it inline because it is narrative voice and quality matters across the scenario.

**vitals / signs / labs:** Uniform shape. Each item has id, label, value (with optional unit), bad flag, _slotRef, and why: null placeholder. Haiku writes the why text.

Example vital:
{
"id": "hr",
"label": "HR",
"value": "168",
"unit": "bpm",
"bad": true,
"_slotRef": "phase[0].vitals.hr.why",
"why": null
}

**actions.tools / actions.meds:** Keyed by ID for fast lookup at render time. Each entry has id, label, priority, _slotRef, and fb: null placeholder. The priority field carries pack-aware logic ("tied-correct", "correct", "distractor-clinical", etc.) and Haiku reads it to write context-appropriate fb text — fb explains both why correct actions are correct and why distractors are wrong.

Example action:
"ioAccess": {
"id": "ioAccess",
"label": "IO Access",
"priority": "tied-correct",
"_slotRef": "phase[0].actions.tools.ioAccess.fb",
"fb": null
}

**Why why: null and fb: null instead of omitting the fields?** Two reasons: it makes the schema self-documenting (you can see what slots exist by inspection), and it lets collectMissingSlots do a simple === null check rather than a hasOwnProperty dance.

## Slot reference scheme

Format:
<container>[.<index>].<collection>.<id>.<field>

Examples:
phase[0].vitals.hr.why
phase[0].labs.wbc.why
phase[0].actions.tools.ioAccess.fb
phase[1].actions.meds.ceftriaxone.fb
debrief.physiologyDeepDive.shock.content

The brackets-with-index notation phase[0] is parsed once at slot-resolve time. The slotResolve.js module already handles dotted paths from Phase 5.2.5 — it needs a minor extension to handle the [N] index notation.

This is the contract Haiku writes against. Every Haiku call gets:

- The slot ref (so it knows where the result will land)
- The slot's local context (the lab name, value, ref range, bad flag)
- The surrounding context (sibling slots, phase-level narrative, top-level presentation summary) — Haiku reads enough context to ensure outputs make sense as a coherent whole, not just slot-by-slot
- For action slots, the priority field so fb text is appropriately framed for correct vs distractor actions

Haiku returns plain text. The dispatcher writes it to the slot via slotResolve.

## Debrief structure
{
"summary": "...",
"keyTeaching": [ ... ],
"physiologyDeepDive": [ ... ],
"performance": { ... }
}

**physiologyDeepDive:** Array (not keyed object) because order matters — Sonnet decides which deep dives to include and what order to present them. Each entry has id, title, _slotRef, content: null. Haiku fills in content.

Example deep dive entry:
{
"id": "septic-shock-pathophys",
"title": "Pathophysiology of distributive shock",
"_slotRef": "debrief.physiologyDeepDive.septic-shock-pathophys.content",
"content": null
}

**summary, keyTeaching, performance:** Sonnet-authored inline (no slots). These are judgment calls Haiku should not make alone — assessing what the user did vs what they should have done is core clinical reasoning that requires the orchestrator's full context.

## Fan-out wave priority

The dispatcher fires Haiku calls in this order:
Wave 1: phase[0].vitals..why, phase[0].signs..why, phase[0].labs..why
Wave 2: phase[0].actions.tools..fb, phase[0].actions.meds..fb
Wave 3: phase[1].vitals..why, phase[1].signs..why, phase[1].labs..why
Wave 4: phase[1].actions.tools..fb, phase[1].actions.meds..fb
Wave 5: debrief.physiologyDeepDive.*.content

Each wave's calls fire in parallel. Next wave starts when previous wave's promises resolve. Wave 1 starts immediately on skeleton return, while user is still on the EMS-report screen.

If the user outruns the queue (clicks into Phase 1 actions before Wave 2 finishes), the slot is rendered with a brief loading shimmer until its specific Haiku call resolves. Same fallback behavior pattern as Phase 5.3.

## Curveball handling (deferred to post-v1)

v1 ships without curveball. This section captures the design constraint so the schema does not paint us into a corner.

**Design rule (locked):** A curveball must be relevant to the disease process, a side effect of an administered medication or intervention, or a meaningful change in body system status. Random unrelated complications are not acceptable.

**Implication 1:** Curveballs split into two categories:
- Static curveballs — disease-process complications that Sonnet can plan upfront (e.g., "deterioration despite appropriate treatment," "septic shock progressing to MODS")
- Dynamic curveballs — response-driven complications that need to react to user choices (e.g., "anaphylaxis to administered antibiotic," "pneumothorax from intubation attempt")

**Implication 2:** Static curveballs can be planned in the initial Sonnet skeleton call. Dynamic curveballs likely require a second Sonnet skeleton call mid-play, after the triggering user action, to plan the appropriate response. The architecture supports this — the second call uses the same orchestrator-worker pattern, just smaller and triggered later.

**Implication 3:** The phase object will need an optional curveballPlanning field with values "static" or "dynamic". v1 implementation can ignore this field; it exists in the schema spec to reserve the namespace.

**Decision deferred to curveball implementation phase:** exact trigger conditions, second-call prompt design, mid-play UX (how the curveball reveals to the user).

## Open questions resolved

This document reflects the resolved design decisions from the Phase 5.4.1 review:

- **Q1 — presentation block placement:** top-level (precedes Phase 1)
- **Q2 — narrative authoring:** Sonnet inline (quality preservation)
- **Q3 — distractor justification slots:** same fb slot, Haiku reads priority field for framing
- **Q4 — curveball as separate stage with relevance constraint:** confirmed; constraint captured above
- **Q5 — _slotRef explicit in JSON:** confirmed (~30-40 tokens per slot, worth it for self-documentation and debug clarity)
- **Q6 (added during review) — curveball planning split:** static planning in initial skeleton, dynamic planning in mid-play second call

## What this document is not

It is not a prompt. The Sonnet skeleton prompt comes in 5.4.2 and will be designed against this schema.

It is not an implementation plan. Code paths, file changes, and migration strategy come in subsequent phases.

It is the contract. Subsequent phases must conform to it. If a subsequent phase reveals a flaw in this schema, the schema is amended explicitly and this document is updated, not silently diverged from.
